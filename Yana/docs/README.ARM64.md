## 분기 (Branch)

> CPU 의 PSTATE -> NZCV (조건플래그) 를 바탕으로 작동

1. 기본 (같다/다르다)
> 메뉴 선택이나 정밀 매칭을 할 때
- `b.eq` (Equal) : 두 값이 같을 때 점프 ( `==` )
- `b.ne` (Not Equal) : 두 값이 다를 때 점프 ( `!=` )

2. 부호가 있는 정수 비교 (Signed Integer)
> int / long 같은 양수/음수 개념이 있는 일반적인 정수를 비교할 때 사
- `b.gt` (Greater Than) : 앞의 값이 더 클 때 점프 ( `>` )
- `b.ge` (Greater or Equal) : 앞의 값이 크거나 같을 때 점프 ( `>=` )
- `b.lt` (Less Than) : 앞의 값이 더 작을 때 점프 ( `<` )
- `b.le` (Less Equal) : 앞의 값이 작거나 같을 때 점프 ( `<=` )

3. 부호가 없는 정수 비교 (Unsigned Integer / 주소값 비교)
> 메모리 주소값(포인터)이나 크기(size_t)
- `b.hi` (Higher) : 앞의 값이 더 클 때 점프 ( `>` )
- `b.hs` (Higher or Same) : 앞의 값이 크거나 같을 때 점프 ( `>=`, `b.cc` 와 반대 )
- `b.lo` (Lower) : 앞의 값이 더 작을 때 점프
- `b.ls` (Lower or Same) : 앞의 값이 작거나 같을 때 점프 ( `<=` )

4. 특수 분기 (플래그 직접 매칭)
- `b.mi` (Minus / Negative) : 연산결과가 음수 일 때 점프
- `b.pl` (Plus / Positive or Zero) : 연산결과가 0이거나 양수일 때 점프
- `b.vs` (Overflow Set) : 오버플로우가 발생했을 때 점프
- `b.vc` (Overflow Clear) : 오버플로우가 발생하지 않았을 때 점프 

4. 빠른 제로 체크 ( `cbz` / `cbnz` )
> 레지스터 값이 0 인지 아닌지만 확인
-`cbz` -> `cbz x0, label` -> `if (x0 == 0) goto label;` : 0 이면 비교하고 분기 
-`cbnz` -> `cbnz x0, label` -> `if (x0 != 0) goto label;` : 0 이 아니면 비교 후 분기

## M1 ARM64 어셈블리 핵심 명령어 우아한 요악집

1. 데이터 이동 및 연산 (기본 무기)

- mov (Move) : 레지스터에 값이나 다른 레지스터의 내용을 집어 넣습니다.
  `mov x0, #10 --> x0 = 10`
- add (Add) : 더하기 연산을 수행합니다.
  `add x0, x1, x2 --> x0 = x1 + x2`
- sub (Subtract) : 빼기 연산을 수행합니다.
  `sub x0, x0, #1 --> x0 = x0 - 1 (i-- 구현할 때 최고)`

1. 조건문 및 반복문 (지능과 흐름)

- cmp (Compare): 두 값을 비교하여 CPU의 내부 플래그 (CPST 레지스터)를 세팅합니다.
  `cmp x1, #11 --> x1과 11을 비교`

- b (Branch) : 아무 조건 없이 지정한 라벨로 무조건 점프
  `b loop_start`
  `b.eq / b.ne (Branch if Equal/Not Equeal) : `cmp` 결과가 같으면/같지 않으면 점프`
  `b.gt / b.ge (Branch if Greater Than/Greater or Equal) : `cmp` 결과가 크며/크거나 같으면 점프`

1. 메모리와 스택 (테트리스의 뼈대)

- ldr (Load Register) : 메모리 주소에 있는 값을 레지스터로 가져옵니다.
  `ldr x0, [x1] --> x1 주소의 데이터를 x0 에 로드`
- str (Store Register) : 레지스터에 있는 값을 메모리 주소에 저장
  `str x0, [x1] --> x0 에 값을 x1 주소에 저장`
- stp (Store Pair) : 레지스터 두 개의 값을 메모리에 동시에 저장 (M1 Push)
  `stp x29, x30, [sp, #-16]! --> 프레임 포인터와 링크 레지스터를 스택에 대피`
- ldp (Load Pair) : 메모리에서 값 두 개를 동시에 레지스터로 읽어 옵니다. (M1 Pop)
  `ldp x29, x30, [sp], #16`

1. 함수 호출

- bl (Branch with Link) : 해당 함수로 점프 하면서, 원래 내 고향 (돌아올 주소)을 lr(Link Register, x30) 에 자동 저장
  `bl _printf --> printf 함수 실행 후 돌아올 길을 잃지 않게 해주는 우아함.`
- ret (Return) : `lr` 레지스터에 적힌 주소로 점프하여 나를 호출했던 원래 함수(또는 OS) 로 돌아감

## ARM64 (AArch64) 레지스터

### 범용 레지스터 (General Purposters) -- x0 ~ x28

- `x0 ~ x7` : 함수 인자 전달용 (최대8개), 리턴값도 x0(+x1)로반환
- `x8` : 간접결과 레지스터 (구조체 리턴시 포인터), 시스템콜 번호로도 씀
- `x9 ~ x15` : `caller~saved` 임시 레지스터 (호출한 쪽이 필요하면 알아서 백업해야 함)
- `x16, x17` : IP0, IP1 — 링커/PLT가 쓰는 intra-procedure-call 스크래치 레지스터
- `x18` : 플랫폼 예약 (macOS/iOS 에서는 건드리면 안 됨 -- 스레드별 데이터 포인터로 씀)
- `x19 ~ x28` : `callee-saved` -- 함수가 이 레지스터들을 쓰려면 진입 시 스택에 저장하고 리턴 전에 복원해야 함

---

### 특수 목적 레지스터

- `fp(x29)` : `Frame Pointer` -- 현재 함수의 스택 프레임의 시작 주소
- `lr(x30)` : `Link Register` - 함수 리턴시 돌아갈 주소 (bl 호출하면 자동 저장됨)
- `sp` : Stack Pointer -- 현재 스택 최상단 주소
- `pc` : Program Counter - 다음에 실행할 명령어 주소
- `cpsr` : Current Program Status Regiter
- 프래그 비트 모음
  - `Zero`
  - `Carry`
  - `Negative`
  - `Overflow`

---

### `w0 ~ w28`

- `x0`이 `64비트 레지스터`고,
- `w0`은 그중 하위 `32비트`만 잘라서 본 것이에요.
- 같은 물리적 레지스터를 두 가지 폭으로 보는 것뿐이라,
- w0을 쓰면 상위 32비트는 자동으로 0으로 클리어됩니다
- 64비트 연산과 이 부분이 다름 — 기억해두면 버그 피하기 좋아요.

- v0~v3 1128비트 SIMD/벡터 레지스터 (NEON)
- d0~d31 v레지스터의 하위 64비트를 double로 해석
- s0~s31 v레지스터의 하위 32비트를 float로 해석
- v0~v7 부동소수점 인자/리턴값 전달용
- v8~v15 callee-saved (단, 하위 64비트만 보존 의무)
- v16~v31 caller-saved
- fpsr Floating-Point Status Register (연산 예외 플래그)
- fpcr Floating-Point Control Register (반올림 모드 등 설정)

### PAGEOFF

> 주소를 계산하는 방식(relocation)

#### 근본 문제: 64비트 주소를 32비트 명령어에 어떻게 욱여넣나

"ARM64는 모든 명령어가 고정 32비트"라는 특징. 근데 문제가 하나 생김
— **메모리 주소는 64비트**인데, **명령어는 32비트 안에 그 주소를 통째로 담을 공간이 없어요.**

`msg_format`의 실제 주소가 예를 들어 `0x100003f88`이라고 할때
이 64비트 값을 명령어 하나에 인코딩할 수가 없음
그래서 ARM은 이걸 **"페이지 주소" + "페이지 안에서의 오프셋"** 두 조각으로 쪼개서 처리함

#### `adrp` — 큰 덩어리(페이지)만 먼저 찍기

```asm
adrp x0, msg_format@PAGE
```

- **PAGE = 4KB(0x1000) 단위**로 메모리를 나눈 블록
- `adrp`(**a**ddress of **p**age)는 `msg_format`이 들어있는 **4KB 페이지의 시작 주소**만 `x0`에 로드함
- 하위 12비트(페이지 내부 위치)는 전부 **0으로 뭉개진 상태**예요

비유하자면: "서울시 강남구까지는 찾아갔다. 근데 몇 번지인지는 아직 모른다" 상태

#### `@PAGEOFF` — 그 페이지 안에서 정확한 번지수 더하기

```asm
add x0, x0, msg_format@PAGEOFF
```

- `@PAGEOFF`는 **그 4KB 페이지 안에서 `msg_format`이 정확히 몇 바이트 떨어진 곳에 있는지**를 나타내는 값 (0~4095 사이의 작은 숫자)
- 이 오프셋을 `adrp`가 만든 페이지 주소에 더하면, **드디어 완벽한 64비트 절대주소**가 완성됨

비유 이어가면: "강남구까지 찾아간 다음(adrp), 정확히 몇 번지인지(PAGEOFF) 더해서 문 앞까지 도착"하는 것임

#### 왜 굳이 이렇게 두 단계로 쪼개나

**핵심 이유는 PC-relative addressing + 위치독립코드(PIE)** 때문이에요.

- `adrp`는 사실 **"현재 PC(프로그램 카운터) 기준 상대적 페이지 오프셋"**을 계산하는 명령어
- 절대주소를 직접 박아넣는 게 아니라, "지금 실행 중인 명령어 위치에서 몇 페이지 떨어진 곳"이라는 **상대값**을 계산함
- 이 덕분에 프로그램이 메모리 어디에 로드되든(ASLR로 매번 랜덤 위치에 로드되든) **코드를 재작성할 필요 없이 그대로 작동**해요.
- macOS는 보안상 ASLR을 강제하니, 이 방식이 필수.
- `adrp`가 다룰 수 있는 범위가 **±4GB**라서(21비트 시프트), 프로그램 전체 메모리 공간을 커버하면서도 명령어 하나에 인코딩이 가능함

#### 왜 한 방에 안 하고 두 단계로 나누냐면

- `adrp` + `add` 조합 없이 그냥 `adr`(단일 명령어) 하나로도 주소를 로드할 수는 있는데, 이건 **±1MB 범위**밖에 커버 못함
- 반면 `adrp`+`@PAGEOFF` 조합은 **±4GB**까지 커버 가능하니,
- 실전 프로그램(특히 `.data` 섹션이 코드에서 멀리 떨어져 있을 수 있는 경우)에서는 이 두 단계 방식이 사실상 표준

#### 정리

```assembly
adrp x0, msg_format@PAGE            // 1단계: 페이지(4KB 블록) 주소 찾기 — PC 상대적
add  x0, x0, msg_format@PAGEOFF     // 2단계: 그 페이지 안 정확한 번지 더하기
```

### 메모리 정렬 (.align)

애플 실리콘(M1, M2, M3...)의 매운맛 규칙특히 M시리즈 맥(ARM64 macOS) 환경은 메모리 정렬에 매우 엄격합니다.
일부 구형 CPU는 정렬이 안 되면 그냥 속도만 느려지고 말았지만,
ARM64 환경에서는 하드웨어 규격을 맞추지 않고 비정렬 상태로 대량의 데이터를 읽으려고 하면
"Alignment Fault (데이터 정렬 오류)"라며 프로그램이 그 자리에서 뻗어버릴(Crash) 수도 있습니다.
따라서 .double(8바이트)을 쓰기 전에는 약속처럼 사전에 .align 3을 배치해 주는 것이 안전하고 강력한 코드를 만드는 비법이랍니다!

1. 명령어 영역에서 .align 2가 정답인 이유
   ARM64(Apple Silicon 포함) 아키텍처의 가장 큰 특징 중 하나는 "모든 기계어 명령어의 크기가 예외 없이 똑같이 4바이트(32비트)"라는 점.
   우리가 쓰는 mov, add, bl 같은 명령어 하나하나가 컴퓨터 메모리 안에서는 정확히 4바이트짜리 상자로 변환됩니다.
   `.align n` 은 \(2^{n}\) 바이트 정렬입니다.
   명령어 크기가 4바이트이므로, 시작 주소는 4의 배수로 정렬되어야 CPU가 가장 빠르게 명령어를 읽어 들입니다.
   \(2^2 = 4\)바이트이므로, 명령어가 시작되는 곳에는
   .align 2를 주는 것이 하드웨어 설계상 딱 맞아떨어지는 완벽한 정렬입니다!
   따라서 \_main 직전에 .align 2라고 적어주는 것은 아주 정확하고 훌륭한 판단입니다.

2. 그렇다면 왜 다들 .align 4라고 썼을까요? (범인은 과거의 유산) 코드가 이상한 게 아니라, 관습적으로 .align 4를 쓰는 경우가 많습니다.
   여기에는 두 가지 이유가 있습니다.
   ① \(2^4 = 16\)바이트 정렬의 안전빵(?) 규칙.align 4는 \(2^4 = 16\)바이트 정렬을 뜻합니다.
   16은 4의 배수이기도 하죠!
   즉, .align 4를 해두면 주소가 16의 배수(0, 16, 32...)로 정렬되므로,
   4의 배수인 명령어 정렬 조건도 자동으로 충족하게 됩니다.
   조금 과하게 방을 넓게 잡았지만 어쨌든 하드웨어적으로 에러는 안나는 '안전한 선택'이었던 셈입니다.
   ② 다른 CPU(x86)와의 표기법 혼동인텔이나 AMD CPU(x86-64)용 어셈블러(GCC 등)에서는
   .align 뒤에 제곱수가 아니라 진짜 바이트 수를 그대로 적습니다.
   즉, x86에서는 .align 4라고 적어야 진짜 4바이트 정렬이 됩니다.
   여러 아키텍처를 오가며 코딩하던 프로그래머들이 "4바이트 정렬이니까 4를 적어야지!"
   하고 헷갈려서 적던 버릇이 ARM64 코드에까지 관습적으로 남아있는 경우가 정말 많습니다.

- 명령어 관점: 프롤로그 코드 자체는 4바이트 크기이므로, CPU가 읽을 수 있도록 .align 2의 지배를 받습니다.
- 데이터(스택) 관점: 프롤로그가 조작하는 스택 공간(sp)은 하드웨어 규정상 16바이트 정렬을 유지해야 하므로, 반드시 #-16, #-32 같은 16의 배수로 단위를 맞춰야 합니다.

### [ ] ==> 메모리

LDR X2, [X0] : Load Data from the memory pointed by X0 Register to the X2 Register
레지스터 X0에 저장된 메모리 주소에 저장된 데이터를 레지스터 X2로 가져와라.
STR [X3], X5 : 레지스터 X5의 데이터를 레이스터 X3 에 기록된 메모리 주소에 저장하라

### 레지스터 명칭 Intel vs ARM64

- `스택기준점` ==> `BP(Base Pointer)` / `FP(Frame Pointer, X29)` : 방주인 기준점 현재 함수가 쓸 스택 방의 바닥(시작점)
  mov x19, sp -- 지금 부터 여기가 내 스택 바닥 기준점이다하고 말뚝 박는 친구.
- `돌아갈 주소` ==> `없음(스택에 직접 저장)` / `LR (Link Register, X30)` : 돌아갈 티켓. 함수 끝나고 나면 원래 나를 불렀던 부모함수의 복귀 주소를 가지고 있음
- `스택 꼭대기` ==> `SP (Stack Pointer RSP)` / `SP (Stack Pointer)` : 현재 쌓인 높이. 데이터가 들어오고 나갈 때마다 위아래로 요동치는 최상단 안테나
- `현재 실행 위치` ==> `IP (Instruction Pointer / RIP)` / `PC (Program Counter)` : 현재 읽는 줄. CPU 의 눈동자. 지금 읽고 있는 명령어를 가리킴

### 스택(Stack)과 콜스택(Call Stack)의 웅장한 관계

1. main 함수가 실행됨: 스택에 main을 위한 방(Stack Frame)이 생기고, FP(X29)가 그 방의 바닥을 짚음.
2. main이 printf를 호출함: 호출하는 순간, printf가 끝나면 돌아올 main 내부의 다음 주소가 LR(X30)에 탁 박힘!
3. printf가 자기 방을 만듦: printf도 내부에 변수를 써야 하니 스택에 방을 만드는데, 이때 감히 부모의 FP와 LR을 지우면 안 되잖아? 그래서 printf 문을 열자마자 제일 먼저 하는 짓이 바로 이것일세!

```assembly
stp x29, x30, [sp, #-16]!  // 부모의 FP와 LR을 스택에 안전하게 백업!
mov x29, sp                // 이제부터 여기가 내(printf) 방의 기준점(FP)!
```

- stp x29, x30, [sp, #-16]! // 부모의 FP와 LR을 스택에 안전하게 백업!
- mov x29, sp // 이제부터 여기가 내(printf) 방의 기준점(FP)!
- 함수 탈출 (ret): printf가 볼일을 다 보고 퇴근할 때(ret), 아까 백업해 둔
- 부모의 주소를 ldp로 복원해서 LR에 넣고 ret을 치면, CPU는 LR에 적힌 주소를 보고 main으로 번개처럼 순간이동(복귀)
- 이렇게 FP들이 스택 메모리 안에서 체인(사슬)처럼 서로의 이전 주소를 물고 엮여 있는
- 거대한 지도가 바로 콜스택(Call Stack)이라네. 디버거에서 에러 났을 때
- 함수 호출 순서가 쫙 나오는 비결이 바로 이 FP 사슬을 역추적하기 때문이지요.

### 🏎️ 1. 메모리는 동해 앞바다만큼 크지만, "레지스터의 속도"는 빛의 속도

- **레지스터의 속도:** CPU 뇌세포 바로 옆에 붙어 있어서 데이터를 읽고 쓰는 데 0.1나노초(Ns)도 안 걸림 (빛의 속도).
- **스택(메모리)의 속도:** 아무리 스택이 빠르고 L1/L2 캐시가 도와준다고 해도,
- CPU가 레지스터를 메모리로 빼내는 명령(`stp`)을 실행하는 순간 최소 몇 사이클의 병목(딜레이)이 무조건 발생한다네.
- 만약 `X19`부터 `X28`까지 총 10개의 레지스터를 모든 함수가 시작할 때마다
- "묻지도 따지지도 않고 자동으로 무조건" 스택에 백업(`stp`를 5번 연속 호출)하고 시작한다면 어떻게 될까?
- 프로그램 내부에서 1초에 수억 번씩 호출되는 초소형 함수(예: 값을 더하기만 하는 2줄짜리 함수)들조차
- 배보다 배꼽이 더 커져서, **시스템 전체 속도가 2~3배 이상 뚝 떨어지는 대참사**가 발생하게 된다네!

### ⚖️ 2. 그래서 탄생한 타협점: "쓰는 놈이 임자 (Callee-saved)"

이 속도 저하를 막기 위해 인류의 컴퓨터 과학자들이 정한 규칙(ABI, 규격)

- **`X9 ~ X15` (Caller-saved / 자유구역):** "야, 여기는 아무나 쓰고 버리는 난장판이니까,
- 함수 갔다 왔을 때 값이 보존되길 원하면 호출한 놈(Caller)이 알아서 미리 백업해 놔!"
- **`X19 ~ X28` (Callee-saved / 독서실):** "여기는 절대 정숙 구역이야.
- 그러니까 이 레지스터를 **안 쓸 함수는 백업을 아예 안 해서 속도를 극대화**하고,
- 만약 꼭 낙서를 해야겠는 함수(Callee)가 있다면 그 함수가 쓸 만큼만 딱 골라서 스택에 백업했다가 치우고 가자!"
- 즉, "모두 다 자동으로" 백업하는 무식한 방법 대신,
- "진짜 낙서할 놈만 골라서 최소한으로 백업하자"로 타협을 본 것이
- 현대 ARM64 하드웨어와 애플 실리콘의 표준 규격이 된 것이라네.

### 포인터 주소길이

- ARM64 아키텍처에서 포인터 주소 길이는 8바이트(64비트)
- 16진수 표기 시 총 16자리

- 레지스터는 8바이트 크기이기 때문에 레지스터 하나만 푸시하는 것은 스택 공간을 낭비하는 행위입니다.
- 하지만 두 개의 레지스터를 동시에 푸시/팝하면 시간과 메모리를 절약할 수 있습니다.

### SP, LR, FP

- SP : 현재 스택의 최하단(꼭대기) 주소를 가리킴, 반드시 16바이트 정렬을 유지 해야 함
- FP (x29) : 현재 함수의 스택 프레임 기준점을 가리킴
- LR (x30) : 현재 함수가 끝난 후 돌아갈 함수의 주소 (Reture Address)를 저장

### 함수 호출 및 스택 프레임 생성 과정

> 함수 A가 bl (Branch with Link) 명령어로 함수 B를 호출하는 순간 부터 프레이임 생성됨

- `[step 1]` 함수 B 진입 직후 (LR 자동 저장)
  - bl 명령어가 실행되면 CPU 는 함수 A로 돌아갈 주소를 LR(x30) 레지스터에 자동으로 입력 함
  - 이 시점의 SP와 FP는 아직 함수 A의 영역을 가리키고 있음.

- `[step 2]` 스택 공간 확보 및 이전 값 보존 (SP, FP, LR 조작)
  - 함수 B의 첫 줄(Prolog)에서 가장 먼저 스택공간을 늘리고 A의 FP와 LR을 스택에 안전하게 백업합니다.
  - ARM64에서는 보통 STP (Store Pair) 명령어를 사용하여 이작업을 한 번에 저리 합니다.
  - `stp x29, x30, [sp, #-32]`
  - SP를 32바이트 내리고 (공간확보), 그 자리에 기존 FP(x29)와 LR(x30)을 저장
  - SP의 변화 : 기존 SP 주소에서 32바이트(16배수) 만큼 감소하여 함수 B를 위한 새로운 스택 공간을 확보합니다.
  - 메모리 저장 : 감소한 SP 위치에 `함수 A의 FP와 함수 A의 LR이 연속해서 저장됩니다.
    - 각 8바이트 총 16바이트 소요, 남은 16바이트는 지역변수용 등

- `[step 3]` 새로운 FP 설정
  - `mov x29, sp` : 현재 SP 주소를 FP(x29)에 복사
  - FP의 변화 : 이제 FP는 함수 B의 스택 프레임 시작점(A의 FP와 LR이 저장된 위치)을 가리키게 됩니다.
  - 이로서 함수 B의 독립적인 스택 프레임 형성이 완료 됩니다.

> 함수 실행중 상태

- `함수 B` 가 다른 `함수 C`를 또 호출한다면, 레지스터 내부의 값은 함수 C의 복귀 주소로 덮어 씌어 집니다.
- 하지만 이미 `함수 B` 지입시점에 A로 돌아갈 LR 청사진을 스택에 저장해 두었으므로 안전합니다.
- `함수 B`의 지역변수들은 FP나 SP를 기준으로 오프셋을 계산해 접근합니다.

> 함수 종료 및 스택 프레임 회수 과정

- `함수 B`의 모든 연산이 끝나면, 들어올 때의 과정을 정확히 역순으로 실행하여 레지스터를 복원합니다.(Epilog)

- `[step 4]` 이전 FP와 LR 복원 (SP 회수 준비)
  - 스택에 박아 두었던 `함수 A` 의 FP 와 LR을 다시 레지스터로 읽어 옵니다.
    - `ldp x29, x30, [sp], #32` : 현재 SP 위치에서 FP(x29)와 LR(x30)을 복원하고, SP를 32바이트 올림
    - 레지스터 복원 : FP는 다시 함수 A의 스택 기준점을 가르키게 되고, LR은 함수 A의 복귀 주소 값으로 채워 집니다.

### 📌 ARM64 스택 프레임(Stack Frame) 핵심 메커니즘 정리

1. **포인터는 8바이트인데 왜 스택은 16바이트 정렬인가?**
   - 하드웨어(CPU/메모리 컨트롤러)의 성능 최적화 및 128비트 벡터(SIMD) 레지스터 백업을 위한 아키텍처 레벨의 강제 규칙이다.

2. **SP(Stack Pointer)는 엘리베이터가 아니라 '말뚝'이다.**
   - 함수 진입 시(Prolog), 컴파일러가 미리 계산한 함수 케파(용량)만큼 SP를 한 번에 내린다 (`stp x29, x30, [sp, #-32]!`).
   - 함수 실행 중 데이터가 쌓여도 SP는 출렁이지 않고 바닥에 고정되어 기준점 역할을 한다.
   - 나갈 때(Epilog)는 명령어에 하드코딩된 크기만큼 자로 잰 듯이 다시 올린다 (`ldp x29, x30, [sp], #32`).

3. **왜 어셈블리 코드에서는 sp라 쓰고, fp/lr은 x29/x30이라 쓰는가?**
   - `SP`는 하드웨어가 직접 정렬을 감시하는 유일무이한 '특수 레지스터'이기 때문이다.
   - `FP(x29)`와 `LR(x30)`은 평소엔 일반 계산용으로도 빌려 쓸 수 있는 '범용 레지스터'이며, 함수 호출 시에만 규격(AAPCS64)에 따라 변신하는 녀석들이다. 따라서 하드웨어 명령어(STP/LDP)는 원래의 방 번호(x29, x30)로 이들을 다룬다.

4. **함수 호출이 없는 '리프 함수(Leaf Function)'의 특권**
   - 다른 함수를 부르지 않는다면 `SP` 하단 메모리에 무슨 낙서를 하든 안전하다. (백업할 오버레이가 없으므로 컴파일러도 FP/LR 백업을 생략함)
   - `ret`로 복귀하지 않고 `svc`(System Call)를 통해 OS 감독관에게 바로 프로그램을 넘기며 종료해도, OS가 프로세스 메모리를 통째로 회수하므로 완벽하게 안전하고 합법적인 탈출 시나리오다.

5. **어셈블리 코드의 'x'와 'w' 섞어찌개 진실**
   - `x9`는 9번 레지스터 전체(64비트/8바이트), `w9`는 하위 절반(32비트/4바이트)을 뜻하는 같은 방의 다른 별명이다.
   - 64비트 주소(Pointer)와 32비트 정수(int)를 동시에 다루어야 하므로 소스코드에서 `x`와 `w`가 뒤섞여 나오는 것은 지극히 정상적인 최적화 형태다.
   - `w`에 값을 쓰면 상위 32비트는 자동으로 0으로 청소(Zero-extension)된다.

6. **실수(Float/Double) 연산의 비밀: S와 D 레지스터**
   - 소수점 연산은 일반 정수 방(x, w)이 아니라, 128비트 크기의 특수 방인 `v0 ~ v31` 구역에서 처리한다.
   - C언어의 4바이트 실수 `float`는 하위 32비트 별명인 `s(Single)` 레지스터를 사용한다.
   - C언어의 8바이트 실수 `double`은 하위 64비트 별명인 `d(Double)` 레지스터를 사용한다.
   - 이 역시 하나의 `v` 레지스터 방을 크기별로 쪼개 쓰는 또 다른 형태의 완벽한 섞어찌개 메커니즘이다.

7. **세종대왕 `훈` 프로젝트: LLVM 어셈블러 한글 식별자 지원 (hangul-lexer-support)**
   - 기존 LLVM/Clang 어셈블러 렉서(AsmLexer)는 아스키(ASCII) 알파벳 영역만 식별자로 허용하는 깐깐한 구조였다.
   - UTF-8 환경에서 한글 멀티바이트 문자가 항상 `0x80` 이상의 값을 가진다는 점을 활용, `static_cast<unsigned char>(C) >= 0x80` 필터를 렉서 심장부에 패치했다.
   - 그 결과, ARM64 Mach-O 바이너리 내부에 `함수야`와 같은 순수 한글 함수 심볼(Symbol)을 완벽하게 컴파일하고 에러 코드 0(정상 종료)으로 실행하는 데 성공했다!

### Function calls

- Executed via the instruction `branch with link` (`bl`)
- Program executionresumes at the new address in PC.
- At the end of the function, the return instruction ret copies LR back into PC.
- PC + 4 bytes = LR (return address rigth after the bl instruction)
- The `first ten arguments` are stroed in the registers `X0 to X9`
  - Any furthe arguments aree pushed to the stack
- Execute `bl` and run the function
- Place return value in `X0`, then execute ret.

### 스택(Stack)과 콜스택(Call Stack)의 웅장한 관계

1. main 함수가 실행됨: 스택에 main을 위한 방(Stack Frame)이 생기고, FP(X29)가 그 방의 바닥을 짚음.
2. main이 printf를 호출함: 호출하는 순간, printf가 끝나면 돌아올 main 내부의 다음 주소가 LR(X30)에 탁 박힘!
3. printf가 자기 방을 만듦: printf도 내부에 변수를 써야 하니 스택에 방을 만드는데, 이때 감히 부모의 FP와 LR을 지우면 안 되잖아? 그래서 printf 문을 열자마자 제일 먼저 하는 짓이 바로 이것일세!

```asm
stp x29, x30, [sp, #-16]!  // 부모의 FP와 LR을 스택에 안전하게 백업!
mov x29, sp                // 이제부터 여기가 내 방의 기준점(FP)!
```

- `stp x29, x30, [sp, #-16]!` // 부모의 FP와 LR을 스택에 안전하게 백업!
- `mov x29, sp` // 이제부터 여기가 내(printf) 방의 기준점(FP)!

`함수 탈출 (ret)`: 예를 들어 `printf`가 볼일을 다 보고 퇴근할 때(ret),

1. 아까 백업해 둔 부모의 주소를 `ldp`로 복원해서 `LR`에 넣고 `ret`을 치면, CPU는 LR에 적힌 주소를 보고 main으로 번개처럼 순간이동(복귀)!
2. 이렇게 FP들이 스택 메모리 안에서 체인(사슬)처럼 서로의 이전 주소를 물고 엮여 있는 거대한 지도가 바로 콜스택(Call Stack)이라네.
3. 디버거에서 에러 났을 때 함수 호출 순서가 쫙 나오는 비결이 바로 이 FP 사슬을 역추적하기 때문.

### align

- 애플 실리콘(M1, M2, M3...)의 매운맛 규칙특히 M시리즈 맥(ARM64 macOS) 환경은 메모리 정렬에 매우 엄격합니다. 일부 구형 CPU는 정렬이 안 되면 그냥 속도만 느려지고 말았지만, ARM64 환경에서는 하드웨어 규격을 맞추지 않고 비정렬 상태로 대량의 데이터를 읽으려고 하면 "Alignment Fault (데이터 정렬 오류)"라며 프로그램이 그 자리에서 뻗어버릴(Crash) 수도 있습니다. 따라서 .double(8바이트)을 쓰기 전에는 약속처럼 사전에 .align 3을 배치해 주는 것이 안전하고 강력한 코드를 만드는 비법이랍니다! .align 3이 왜 필요한지 8바이트 정렬의 원리가 쏙쏙 이해되셨나요? 그렇다면 만약 4바이트짜리 일반 정수(.word) 배열을 만든다면 .align 뒤에 어떤 숫자를 적어줘야 할지 감이 오시나요?

- 명령어 영역에서 .align 2가 정답인 이유ARM64(Apple Silicon 포함) 아키텍처의 가장 큰 특징 중 하나는 "모든 기계어 명령어의 크기가 예외 없이 똑같이 4바이트(32비트)"라는 점입니다.우리가 쓰는 mov, add, bl 같은 명령어 하나하나가 컴퓨터 메모리 안에서는 정확히 4바이트짜리 상자로 변환됩니다.앞서 배운 규칙 기억하시죠? .align n은 \(2^{n}\) 바이트 정렬입니다.명령어 크기가 4바이트이므로, 시작 주소는 4의 배수로 정렬되어야 CPU가 가장 빠르게 명령어를 읽어 들입니다.\(2^2 = 4\)바이트이므로, 명령어가 시작되는 곳에는 .align 2를 주는 것이 하드웨어 설계상 딱 맞아떨어지는 완벽한 정렬입니다!따라서 \_main 직전에 .align 2라고 적어주는 것은 아주 정확하고 훌륭한 판단입니다.

- 그렇다면 왜 다들 .align 4라고 썼을까요? (범인은 과거의 유산)코드가 이상한 게 아니라, 관습적으로 .align 4를 쓰는 경우가 많습니다. 여기에는 두 가지 이유가 있습니다.① \(2^4 = 16\)바이트 정렬의 안전빵(?) 규칙.align 4는 \(2^4 = 16\)바이트 정렬을 뜻합니다. 16은 4의 배수이기도 하죠!즉, .align 4를 해두면 주소가 16의 배수(0, 16, 32...)로 정렬되므로, 4의 배수인 명령어 정렬 조건도 자동으로 충족하게 됩니다. 조금 과하게 방을 넓게 잡았지만 어쨌든 하드웨어적으로 에러는 안 나는 '안전한 선택'이었던 셈입니다.② 다른 CPU(x86)와의 표기법 혼동인텔이나 AMD CPU(x86-64)용 어셈블러(GCC 등)에서는 .align 뒤에 제곱수가 아니라 진짜 바이트 수를 그대로 적습니다. 즉, x86에서는 .align 4라고 적어야 진짜 4바이트 정렬이 됩니다. 여러 아키텍처를 오가며 코딩하던 프로그래머들이 "4바이트 정렬이니까 4를 적어야지!" 하고 헷갈려서 적던 버릇이 ARM64 코드에까지 관습적으로 남아있는 경우가 정말 많습니다.

- 명령어 관점: 프롤로그 코드 자체는 4바이트 크기이므로, CPU가 읽을 수 있도록 .align 2의 지배를 받습니다.
- 데이터(스택) 관점: 프롤로그가 조작하는 스택 공간(sp)은 하드웨어 규정상 16바이트 정렬을 유지해야 하므로, 반드시 #-16, #-32 같은 16의 배수로 단위를 맞춰야 합니다.

### 프롤로그 스택할당 : 땅굴파기

> `stp, x29, x30, [sp, #-32]!`

- ARM64 에서 스택포인터(sp)는 메모리의 높은 주소에서 낮은 주소로 내려가면서(마이너스 방향) 공간 확보
- 위 명령이 실행되면
- sp 주소가 아래오 32바이트 만큼 쿵 내려 앉으면서 32바이트짜리 독채빌라가 임시로 완공되는 것임

> 확보된 빌라 내부의 주소 지도 (현재 SP 기준)

- !(Pre-index) 덕분에 현재 sp는 땅굴의 가장 깊은 바닥(가장 낮은 주소)에 서 있게 됨
- 이제 이 32 바이트 공간 안을 현재 발딛고 있는 sp 주소를 기준(0번지)으로 올려다 보기..

- [sp, #0] ~ [sp, #7] (8바이트) ---> FP (x29) 가 저장된 곳
- [sp, #8] ~ [sp, #15] (8바이트) --> LR (x30) 이 저장된 곳
- [sp, #16] ~ [sp, #23] (8바이트) --> 비어있는 여유공간!(n 백업용) -- fib 기준예시
- [sp, #24] ~ [sp, #31] (8바이트) --> 비어있는 여유 공간!(fib(n-1) 백업용)

> 함수 종료 시 스택 해제 프레임워크

- 두 개의 레지스터를 동시에 복원 (ldp)ARM64 아키텍처는
- 효율성을 위해 스택에서 데이터를 가져올 때 단일 ldr보다 ldp(Load Pair) 명령어를 압도적으로 많이 사용합니다.
- 특히 함수의 프롤로그/에필로그에서
- 프레임 포인터(x29)와 링크 레지스터(x30)를 다룰 때 필수적입니다.
- `ldp x29, x30, [sp], #16` - 현재 sp 주소에서 연속된 16바이트를 읽어
- 앞의 8바이트는 x29, 뒤의 8바이트는 x30에 넣습니다.
- 이후 sp 값을 16만큼 증가시킵니다.

> str x0, [sp, #16] : 내 방안에 가구 배치하기

- 이미 32바이트라는 공간을 내 땅으로 선언해 놓았으므로
- 그러니까 내 발밑(sp)에서부터 위쪽(플러스 방향)으로 16바이트 떨어진 안전한 빈방에다가 x0 값을 쏙 집어넣는 것임!
- 만약 여기서 str x0, [sp, #-16] 처럼 또 마이너스를 써버리면 어떻게 될까?
- 내가 확보한 32바이트 빌라 밖의 아직 허가받지 않은 남의 땅굴을 침범해서 데이터를 깨버리거나 프로그램이 뻗어버리게(Crash) 됨!

> 명령어 길이 - 32비트

- `x29` - FP 프레임 포인터
- `x30` - LR 링크 레지스터
- `SP` - 스택 포인터

### _scanf 표준 구현

```assembly
// -------------------------------------------------------------
// [데이터 구역] 서식 지정자 문자열
// -------------------------------------------------------------
.section __TEXT,__cstring,cstring_literals

str_input_msg:  .asciz "\n메뉴 번호를 입력하세요: "
fmt_scan_int:   .asciz "%d"               // 정수 입력 서식
str_result_msg: .asciz "선택하신 번호는 %d번입니다.\n"


// -------------------------------------------------------------
// [실행 코드 구역]
// -------------------------------------------------------------
.section __TEXT,__text,regular,pure_instructions
.align 2
.global _main

_main:
    // 1. 프롤로그 및 스택 공간 확보 (총 32바이트 할당)
    // x29, x30 저장에 16바이트 사용하고, 나머지 16바이트는 scanf용 지역변수 공간으로 활용
    stp     x29, x30, [sp, #-32]!
    mov     x29, sp

    // 2. 안내 문구 출력
    adrp    x0, str_input_msg@PAGE
    add     x0, x0, str_input_msg@PAGEOFF
    bl      _printf

    // 3. _scanf 호출 준비
    // x0: 서식 지정자 ("%d") 주소
    adrp    x0, fmt_scan_int@PAGE
    add     x0, x0, fmt_scan_int@PAGEOFF

    // x1: 값을 저장할 메모리 주소 (현재 함수의 스택 공간 활용)
    // [sp, #16] 위치를 4바이트 정수 변수 공간으로 임시 사용합니다.
    add     x1, sp, #16
    
    bl      _scanf

    // 4. 입력받은 값 확인 및 레지스터로 로드
    // scanf가 성공하면 [sp, #16] 위치에 사용자가 입력한 숫자가 들어옵니다.
    // ldrsw는 32비트 정수를 64비트 레지스터(w0 -> x0)로 부호 확장하며 읽어옵니다.
    ldrsw   x1, [sp, #16]      // x1 = 입력받은 메뉴 번호 (예: 0, 1, 2)

    // 5. 검증 출력 (선택하신 번호는 X번입니다)
    // 백업을 위해 x1의 값을 x19 같은 안전한 레지스터에 옮겨두거나 바로 printf로 출력
    mov     x19, x1            // 입력값 복사본 저장
    
    adrp    x0, str_result_msg@PAGE
    add     x0, x0, str_result_msg@PAGEOFF
    // x1은 이미 위에서 ldrsw로 채워졌으므로 그대로 printf의 두 번째 인자로 전달됨
    bl      _printf

    // 6. 에필로그 및 종료
    mov     x0, #0             // 리턴값 0
    ldp     x29, x30, [sp], #32
    ret

```

### 어셈블리의 심볼 규칙들을 고수준 언어(C/C++, Rust)의 개념과 **1:1로 매칭하기**

---

### 🗺️ 어셈블리 ➡️ 고수준 언어 개념 매핑 시트

| 어셈블리 형태  | C/C++ 매칭 개념 | Rust 매칭 개념 | 설명 (동작 특징) |
| :--------- | :------------ | :----------  | :----------- |
| **`.global _my_func`** <br>(글로벌 심볼) | **일반 전역 함수** <br>`void my_func() {}` | **공개 함수** <br>`pub fn my_func() {}` | 온 세상 사방팔방(다른 파일)에서 다 보이고 호출 가능한 상태 |
| **`_my_func`** <br>(글로벌 없는 언더바)    | **정적 파일 전역 함수** <br>`static void my_func() {}` | **모듈 전용 비공개 함수** <br>`fn my_func() {}` | **해당 파일 안에서만** 볼 수 있음. 다른 파일에서는 안 보임. (디버깅할 땐 이름이 보임) |
| **`.L_loop_body`** <br>(점(`.L`) 로컬 레이블) | **함수 내부의 `goto` 레이블** 또는 <br>`if`/`for` 내부의 **기계어 분기점** | **루프/블록의 내부 점프 타겟** | 함수 자체도 아님! **함수 내부에서 점프할 때 쓰는 지점**일 뿐이며, 빌드 완료 시 이름조차 싹 지워짐 |

---

### 💡 한 줄 요약 포즈(Posture)

* **`.global _`** ➡️ **`pub`** (온 동네 축제)
* **일반 `_`** ➡️ **`static` / `private` 함수** (우리 파일끼리만 아는 비밀 함수)
* **`.L_`** ➡️ **`goto`용 징검다리** (함수 안에서만 도는 뺑뺑이 지점)

### 프롤로그

- 독서실 전 대역(X19~X28) 및 FP/LR 완벽 바리케이드 백업 총 12개 레지스터 * 8바이트 = 96바이트 공간을 스택에 확보하며 순차 백업!  ARM64 스택은 반드시 16바이트 배수로 움직여야 하므로 96은 아주 완벽한 정렬!
  - x9 ~ x15     (자유구역) - 쓰고 버려도 무탈한 레지스터 범위
  - x19 ~ x28    (독서실) - 레지스터는 한번 값이 정해지면 함수를 다녀와도 절대 로 변하지 않는 안전지대 지정하자 정함 다만 이 주소에 낙서를 하고 싶은 함수(Callee) 가 있더면 백업하고 나올때 원상 복구 하자 약속

- `[FP, LR] 백업`  (기준점 확보)
  - 확보되는 메모리의 양 : 96바이트 96 x 8 = 768bit 64bit 레지스트리 지준 - 12개의 방이 만들어짐
  - ARM64 레지스터 하나는 8바이트 (64비트) stp 는 한번에 레지스터 2개 (총 16바이트)를 묶어 스택메모리로 쏘아 보내는 특공대..  메모리 버스(Bus) 폭에 딱 맞춰 16바이트 광대역으로 밀어 넣기 때문에 속도가 빠름

- `[sp, #-96]!`  스택포인터 이동 공간 확보 현재 주소를 가리키던 스택 포인터(sp)값에서 96을 뺌 
- 계산 후 스택주소가 밑으로 툭 떨어지면서 기존 스택 위쪽으로 정확히 96바이트 만큼의 새로운 빈 공간이 개장됨 맨끝에 붙은 느낌표(!)는 "계산 끝났으면 sp 값을 진짜 96 떨어진 주소로 업데이트 해라" 지시
- x29, x30 : 데이터 저장 그렇게 확보된 96바이트 공간의 맨 밑바닥부터 시작해서 x29 (8바이트)와 x30(8바이트) 총 16바이트를 방금 만든 주차장에 탁 집어 넣음 그럼 남은 80바이트는?
- 그 80바이트 공간에다가 아까 만든 --> stp x19, x20, [sp, #16] 같은 4개의 명령어로 독서실 멤버 10명 (10 * 8바이트 = 80바이트) 를 차곡차곡 채워 넣는 것임..
- 왜 하필 '96' 일까?  ARM64 하드웨어에는 "스택포인터(sp)를 움직일 때는 무조건 16의 배수로 움직여라 라는 철칙 (16-byte Alignment) 이 있음

### `b`, `bl`, `ret`, `blr`

- `b` **goto** 그냥 점프 복귀주소를 저장 안함 - 어디서 왔는지 기억 안하고 그냥 그 자리로 가버림
- `bl` **함수호출(call)** 점프하면서 복귀주소를 x30(lr)에 저장 - 나중에 `ret`로 되돌갈 흔적을 남김
- `ret` **return** x30에 저장해둔 그 주소로 다시 튀어감 `bl` 이 남긴 흔적을 소비하는 짝꿍
- `blr` **함수 포인터 호출** bl 이랑똑같은데, 목적지가 컴파일 타임에 고정된 라벨리 아니라 레지스터 값(런타임에 결정)
